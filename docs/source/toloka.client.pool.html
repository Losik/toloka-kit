A set of tasks that are issued and checked according to the same rules within the project

Groups tasks by the following criteria: one-time start-up, which performers can perform tasks, quality control, price for TaskSuite’s, overlap. Tasks, golden tasks and assignments are related to a pool.

For pools that don’t use “smart mixing”. Assign task suites in the order in which they were uploaded. For example, for a pool with an overlap of 5, the first task suite is assigned to five users, then the second task suite, and so on. class toloka.client.pool.dynamic_overlap_config.DynamicOverlapConfig(*, type: Optional[toloka.client.pool.dynamic_overlap_config.DynamicOverlapConfig.Type] = None, max_overlap: Optional[int] = None, min_confidence: Optional[float] = None, answer_weight_skill_id: Optional[str] = None, fields: Optional[List[toloka.client.pool.dynamic_overlap_config.DynamicOverlapConfig.Field]] = None)¶

Dynamic overlap setting.

Allows you to change the overlap depending on how well the performers handle the task. Set the closing interval (auto_close_after_complete_delay_seconds). It should be enough to complete tasks with an overlap higher than the minimum. When all pool tasks are completed, aggregate the responses.

type¶

The algorithm for dynamic overlap.

Type

Optional[Type]

max_overlap¶

Maximum overlap. Must be higher than the values in defaults. Minimum — 1. Maximum — 30000.

Type

Optional[int]

min_confidence¶

Minimum confidence of the aggregated response. Values from 0 to 1.

Type

Optional[float]

answer_weight_skill_id¶

A skill that determines the weight of the performer’s response. For best results, use a skill calculated as percentage of correct responses in control tasks.

Type

Optional[str]

fields¶

Output data fields to use for aggregating responses.

Type

Optional[List[Field]]

class Field(name: Optional[str] = None)¶

Output data fields to use for aggregating responses.

For best results, each of these fields must have a limited number of response options. Don’t specify several fields if their values depend on each other.

name¶

The output data field name.

Type

Optional[str]

class Type(value)¶

The algorithm for dynamic overlap.

BASIC — Each response is assigned a weight depending on the performer’s skill value.

The aggregated response confidence is calculated based on the probability algorithm. The task overlap increases until it reaches max_overlap or until the confidence of the aggregated response exceeds min_confidence.

You have to specify max_overlap, min_confidence, answer_weight_skill_id and fields.

BASIC = 'BASIC'¶

Type

Optional[int]

Minimum number of training tasks in a task suite (if the number of assignments left is less than the one specified in golden_tasks_count). Minimum — 0. By default, the value is the same as in training_tasks_count.

Type

Optional[int]

force_last_assignment¶

Setup for the last set of tasks in the pool, if less than the minimum remaining number of tasks are not completed (mixer_config.min_real_tasks_count). Values: * true - issue an incomplete task set. * false - don’t issue tasks. This option can be used if you are adding tasks after the pool is started. This parameter only applies to main tasks. The number of control and training tasks in the last set must be complete (golden_tasks_count, training_tasks_count).

Type

Optional[bool]

force_last_assignment_delay_seconds¶

Waiting time (in seconds) since the addition of the task, or increase in the overlap, prior to the issuance of the last set of tasks in the pool. The minimum is 0, the maximum is 86,400 seconds (one day). This parameter can be used if the pool has force_last_assignment: True.

Type

Optional[int]

The order for including tasks in suites: * True — Add tasks to suites in the order in which they were uploaded. For example, in a pool with an

overlap of 5, the first uploaded task will be included in the first 5 task suites. They will be assigned to 5 users.

  • False — Add tasks to suites in random order.

Type
